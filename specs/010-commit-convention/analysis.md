# Analysis: Configurable Commit Convention

## Summary

The spec → plan → tasks set is internally consistent and faithful to the real codebase. The coverage matrix closes (all 6 FRs → plan components C1–C7 → tasks T001–T008; all 5 SCs → test-bearing tasks; all 4 user stories → phases), the file mapping was verified against the actual installer (`installScripts` reads `scripts/bash/`, `templateSource` reads `templates/`, both confirmed present with the existing advisors' sources), and no constitution principle is violated. There are **no critical issues** — nothing blocks implementation. The findings are all advisory: test-coverage gaps for explicitly-stated requirements, one under-specified task detail (the new git-running advisor must scope to the target repo), and a scope-clarity note about an extension that lives in plan/tasks but not as an explicit spec FR. These should be absorbed during implementation (as spec 009 absorbed its advisories).

## Issues Found

### Critical

None. The design is sound and every FR has both a plan component and a task. Verified specifically:
- **No FR without a task.** FR-001→T002/T003, FR-002→T002, FR-003→T004, FR-004→T005, FR-005→T007, FR-006→T006.
- **No install-path break.** `installScripts` (utils.ts:293) copies `scripts/bash/*.sh`; `scripts/bash/` exists at repo root with the existing advisors' sources, so `scripts/bash/verify-commit.sh` (plan C3 / task T005) will install correctly. `templateSource` (utils.ts:73) reads `templates/`; `templates/` exists with the existing template sources, so `templates/convention-template.md` (plan C1 / task T002) will seed correctly.
- **No constitution violation** (see Constitution check under Advisory/Positive) and **no dependency cycle**.

### Advisory

1. **A1 (highest) — "honors a custom type set" (FR-004 / SC-003) has no test case.** FR-004 requires the advisor to "honor a custom type set / scope format declared in the convention (not a hardcoded list)", and SC-003 asserts it. But task T005's listed RED cases (`feat(x): y`→conforming, `T001: foo`→flagged, merge→skipped, ABSENT, TEMPLATE) all use the **default** type set — none writes an AUTHORED `.spec/convention.md` declaring a *custom* `allowed_types` (e.g. `build deploy`) and asserts the advisor accepts `build: foo` while flagging `feat: foo`. The plan C3 *designs* custom-set parsing ("parse the convention's allowed_types line"), so a faithful implementer will build it — but without a test, a hardcoded-default implementation would pass the listed cases and silently violate FR-004/SC-003. **Recommendation:** add one custom-`allowed_types` test case to T005.

2. **A2 — FR-001's "update never overwrite" sub-clause is untested.** FR-001 says "init/**update** MUST never overwrite an AUTHORED convention". T002's never-clobber test covers `init` over an AUTHORED convention. `update` (update.ts:17) satisfies the clause *by absence* — it calls `installDocumentTemplates` (refreshes `.spec/templates/convention-template.md`, the regenerable template) but never `installConventionToMemory`, so `.spec/convention.md` is untouched. Correct, but untested; a future change to `update` could regress it. spec 009 locked the analogous invariant for *both* init and update (its T007). **Recommendation:** add a lock test that `runUpdate` over an AUTHORED `.spec/convention.md` preserves it (mirror spec 009's never-clobber test), to T002 or T007.

3. **A3 — `verify-commit.sh` must scope git to the target repo (`git -C "$REPO_ROOT"`); T005's description omits this.** This is the **first spec-coach advisor that runs `git`** (`verify-constitution-sync.sh` and `verify-spec.sh` only grep files). T005 specifies `git log -1 --format=%s HEAD` without repo-scoping. In the headless test the advisor is driven via `execSync` from the *test's* cwd (the repo), against a fixture `mkdtemp` git repo — bare `git log` would read the repo's HEAD, not the fixture's, and the test would fail for the wrong reason (or pass misleadingly if cwd happens to match). `get_repo_root` resolves the fixture, so the fix is `git -C "$REPO_ROOT" log …` (or `cd`). TDD will catch a bare-`git log` bug, but the task should state it so the implementer gets it right first try. **Recommendation:** T005 should specify `git -C "$REPO_ROOT"`.

4. **A4 — Convention preserve-on-uninstall (task T003) has no explicit spec FR.** FR-001 declares the convention "charter-as-IP tier, like the constitution" and extends the spec 009 *never-clobber* (init/update) invariant — but it does **not** explicitly state preserve-on-uninstall (spec 009 had that as a *separate* FR-006). The plan/tasks introduce uninstall status-aware preserve (T003, `isAuthoredConvention`) as a "charter-as-IP extension". It is *implied* by US1 acceptance #3 ("survives any spec-coach command" — uninstall is a spec-coach command) and is the coherent thing to do (otherwise an authored `.spec/convention.md` is orphaned on uninstall). But because it rests on implication rather than an explicit FR, the spec is slightly under-specified here. **Recommendation:** either confirm US1 #3 "any command" as the intended hook (fine), or add an explicit FR (e.g. FR-001 addendum) for preserve-on-uninstall so the requirement is unambiguous. Not blocking — T003 implements it either way.

5. **A5 — T007's guardrail must delimit the four state functions precisely; the mechanism is left to implementation.** T007 asserts `resolve_feature`/`infer_phase`/`first_pending_task`/`get_feature_paths` do not parse `git log` commit subjects, scoped to avoid false-positiving on `verify-commit.sh`'s legitimate `git log`. The intent is clear and testable, but the actual mechanism (extract each function's body — e.g. `awk` between the function header and the next function / closing brace — *then* grep for `git log … %s`) is unspecified. A naive repo-wide grep would false-positive on `verify-commit.sh`. **Recommendation:** the implementer should extract function bodies (not grep the whole file) — state this in T007.

6. **A6 (low) — T006's dependency on T004 is narrative, not technical.** T006 (dogfood `.spec/convention.md` + constitution amendment) is marked dependent on T002 (template) **and** T004 (skills reference convention.md). The T004 dependency is "dogfood closure" coherence, not a technical prerequisite — T006 authors `.spec/convention.md` from the T002 template and could technically run after T002 alone. Marking it serial after T004 is *safe* (conservative) but slightly over-serializes. No action required; noted for transparency.

### Constitution Check (per principle — no violations)

- **I Markdown-Is-the-Product** ✓ — the convention is a markdown doc + a bash advisor; the only new TS is `installConventionToMemory` mirroring the existing `installConstitutionToMemory`. No coaching orchestration code.
- **II Coach-Not-Gatekeeper** ✓ — `verify-commit.sh` always exits 0; no commit-msg hook (explicitly rejected in plan).
- **III Zero-Dependencies** ✓ — verified: the advisor is pure bash (`grep`/`sed`/`git`); **no new npm dependency** is added (commitlint/husky rejected). Template + script are files, not deps.
- **IV Precision in Templates** ✓ (claimed) — `convention-template.md` to use RFC 2119 + a documented machine-readable rules block; the implementer must honor this (test via content-assertion).
- **V Verify What Ships** ✓ — content-assertion + `execSync` advisor tests + installer lock tests.
- **Versioning**: MINOR 2.4.0→2.5.0 is correct — adding `convention-template` + `verify-commit.sh` is **additive** to the install contract (new files in the existing `templates/` + `scripts/bash/` locations named in the constitution's file-structure clause); it does not break `update`, so not MAJOR. `.spec/convention.md` is a new managed artifact analogous to the existing `.spec/memory/constitution.md`, not a contract break.

- **File mapping verified against real code** — `scripts/bash/` and `templates/` source dirs confirmed present with the existing advisors'/templates' sources, so the two new files install through the existing `installScripts` / `installDocumentTemplates` / `installConventionToMemory` paths with no new installer machinery (the convention installer is a faithful mirror of the constitution installer, including the never-clobber guard).
- **Naming is drift-free across plan ↔ tasks** — `installConventionToMemory`, `isAuthoredConvention`, `verify-commit.sh`, `.spec/convention.md`, and the signature-token set (`[PROJECT_NAME]`/`[ALLOWED_TYPES]`/`[SCOPE_FORMAT]`) are identical in both.

- **FR-005 boundary is correctly framed** — commit-format-as-not-a-state-source is the right call given spec 008, and the guardrail's scoping (the four state functions, not a blanket `git log` ban) correctly avoids forbidding the advisor itself.
