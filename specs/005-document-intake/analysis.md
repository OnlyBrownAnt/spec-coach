### Advisory

- **A1 — SC-003 is only half-verifiable headlessly.** SC-003 ("AI-transform produces a `specs/NNN-slug/spec.md` that conforms to the spec template, coached by `spec-absorb`, with zero transformation code in the CLI") has two halves. The "zero transformation code in the CLI" + "staging marks `absorb-ai-pending`" halves are headlessly testable (T012 greps `src/commands/intake.ts` and asserts the staging behavior). The "produces a conformant spec" half is **not** — the transform is performed by the AI via the skill, which is `npm test` (AI-driven) territory and cannot run headless here (per project memory). *Absorb in implementation*: T012 should assert only the verifiable halves and explicitly note (in a test comment + the close-out report) that coaching quality is left to the AI-driven suite, not claimed from the mechanical run. No spec change needed — just an honest scope statement.

- **A2 — `absorbed-ai` status is declared but never reached by any task.** `CandidateStatus` (T002/plan locked signatures) includes `"absorbed-ai"`, and the plan's C5 says "a follow-up `runIntakeProcess`/scan marks the entry `absorbed-ai`." But no task implements the `absorb-ai-pending → absorbed-ai` transition — there is no defined trigger (the skill writes the spec out-of-band; the CLI has no task that detects the written artifact and flips the status). This is non-blocking: T005's idempotency preserves `absorb-ai-pending` entries (they are in the "keep, don't re-surface" set), so re-scans stay clean regardless. *Absorb in implementation*: either (a) have `runIntakeScan` flip a `absorb-ai-pending` entry to `absorbed-ai` when its recorded destination `specs/<slug>/spec.md` now exists (record the slug at staging time in T010), or (b) drop `absorbed-ai` from the type and document that AI-absorb is terminal at `absorb-ai-pending`. Recommend (a) — it closes the loop and makes the manifest an honest record.

- **A3 — The "source deleted between scan and process" edge case has no owning task.** The spec lists it ("process reports the missing source, marks the entry `source-missing`, and continues without crashing"); the `source-missing` status exists. T005 handles *scan-time* pruning of vanished sources, but no task covers *process-time* missing sources — `runIntakeProcess` verbatim/ai/ignore (T006/T010/T013) target candidates whose source may have vanished since the last scan. *Absorb in implementation*: T006 (and by extension T010/T013) should, before copying/staging/ignoring a target, check the source exists; if not, set `status = "source-missing"` and continue (no throw). Add a unit case in `intake.test.ts`.

- **A4 — Build-order: T007 cannot wire the `ignore` subcommand dispatch (CLI) — `runIntakeIgnore` is created in T014.** T007 (US1) describes wiring `case "intake"` to dispatch `scan`, `process`, **and `ignore <verb>`**. But `runIntakeIgnore` is not exported until T014 (US3); importing/calling it in T007 would break the build (undefined import) between T007 and T014. Note `process --ignore` is fine to wire in T007 because it routes through `runIntakeProcess` (exists from T006). *Absorb in implementation*: T007 wires `scan` + `process` (all three mode flags, since `runIntakeProcess` grows the modes incrementally); the **`ignore` subcommand dispatch + its import** lands in T014 (or the T015 capstone). Fix the T007/T014 text accordingly — this is the one advisory that touches task descriptions directly.

- **A5 — Candidate path normalization for cross-platform `isIgnored` matching.** `discoverCandidates` (T004) stores `Candidate.path`; `isIgnored` (T003) matches exact or `p + "/"` prefix using `/`. On Windows, `path.relative` yields `\`-separated segments, so prefix matching against `/`-joined patterns would silently fail. The plan's Technical Context claims `node:path` separator safety, but the manifest is a string store that must be consistent across read/write/match. *Absorb in implementation*: normalize candidate paths and ignore patterns to POSIX forward-slashes before writing the manifest/ignore file and before matching (e.g., `rel.split(path.sep).join("/")`). Add a unit case. Low-risk but real for the cross-platform CLI claim.

- **A6 — Migration: `agents update` installing `spec-absorb` is asserted only indirectly.** The spec assumption says existing projects gain `spec-absorb` via `agents update --all`; T011 asserts `agents add claude` installs 12 skills. `runAgentsUpdate` installs via `installAllSkills` (data-driven by `SKILL_NAMES`), so it will pick up the 12th skill automatically — but no test pins it. *Absorb in implementation*: in T011 (or `agents-update.test.ts`), add one assertion that `agents update` on an existing binding installs `spec-absorb` (i.e., `ownedSkillUnits`/on-disk set includes the new unit). Optional but cheap insurance against a future regression that breaks the documented migration path.
